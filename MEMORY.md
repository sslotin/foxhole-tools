## Product Context

Logistical tools for Foxhole (online war game). Users paste CSVs from in-game bases into a web app to track supplies.

**Two base types:**
- **Frontline bases** (~220 lines) — individual items players use. Inventory mode: compact display, delta tracking, zero-count items hidden.
- **Storage depots** (~430 lines) — crates shipped in bulk (60 crates/container, 1-14 containers/vehicle). Stockpile mode: planning, source/target merging, shopping list auto-fill.

Items must be logically grouped (consumption is correlated). Also hosts an advanced logi guide at `public/guide/`.

## Data Flow

### CSV Parsing
```
Paste → App.vue.addCSV(text) → parser/csv-parser.parseCSV(text)
  → Split lines, parse header (hex, subhex, type, owner, coords)
  → For each item line (index 1+): look up position array[i-1] → codeName
  → If stockpile (>400 lines): suffix codeName with "-crated"
  → Return { hex, subhex, stockpileName, stockpileType, coords, items, isStockpile }
```

**Position tables** (`positions-*.js`): arrays mapping 1-based line indices → codeNames. Generated by `generate-positions.mjs` from `metadata.json` + reference CSVs. Imported as ES modules — no `fs` dependency, works in browser. Avoids language-dependent name matching (Russian CSVs parse identically).

| Stockpile | ~430 | Multi-source/target, crate planning, category filters, shopping list auto-fill, state export/import (JSON) |

## Facility Cost Calculator

Resolves the recipe graph bottom-up over the facility recipes and shows **two panels**: **left** = Raw / Intermediate / By-products resources; **right** = recipes grouped by building/modification. The algorithm and all specifics live in source comments — see `src/facility-calc/planner.mjs`, `resolver.mjs`, `recipes.mjs`, `store.mjs`, `activation.mjs` (all unit-tested).

**Algorithm model (`planner.mjs`):** STATE is `codeName → recipe | null` (null = imported). Targets start imported; the planner assigns a recipe to every non-basic, non-imported resource reachable from the desired targets. `BASIC_RESOURCES` (`Metal, Coal, Sulfur, Components, Oil, Energy`) are terminal **while imported** (the "initial pass" stop condition) — but if the user opts out of importing one (`toggleSkipAutoImport`) or assigns it a recipe, it is manufactured with its default recipe (the "break down a basic" exploration). Evaluation runs in a fixed (topological) order; a recipe's by-products feed a supply pool that reduces later demand, so ordering can drive runs to zero; leftover supply is reported as by-products. **Fractional scale** (no integer-run rounding) — but display counts are ceiled via `Math.ceil(n - 1e-6)`.

**`reachableRecipes` (`resolver.mjs`):** stable graph **closure** independent of `selectedRecipes` — so toggling a choice never adds/removes sections, only flips lit/dim. Returns `Map<recipe, primaryOutput>`.

**`activation.mjs` (pure, unit-tested):** a recipe is **activatable** (full-bright, clickable) iff it produces an item in `raw ∪ inputs ∪ intermediate ∪ {Energy}`; the **clickable** set is `activatable ∪ active` (active recipes stay clickable so they can be deactivated). Non-activatable recipes are dimmed (`opacity:0.4`, `pointer-events:none`).

**Key UI / interaction model:**
- **Recipe rows:** 5-column grid (facility icon + tooltip, inputs, arrow, outputs, time-on-active). Clicking an *inactive* recipe **pins** it (`chooseRecipe`); clicking the *active* recipe **deactivates** it AND **force-imports** the resource (it must clear the pin and mark imported, else the override loop re-assigns it). This is **force-add, not a toggle** (a toggle could no-op when import/pin state disagrees).
- **Left-panel resource click** (`handleInputClick`) decides the action from the **current plan assignment** (produced ⇒ unassign + force-import; reducible imported ⇒ assign default recipe; basics ⇒ `toggleSkipAutoImport`; Energy ⇒ `toggleEnergy`) — always exactly one concrete action, never a no-op.
- **Sticky hover focus:** hovering a left row sets `focused`; the right panel filters to that resource's recipe options. Focus only resets on leaving the whole `.fac-calc` root, so options stay visible while the mouse travels into the right panel.
- **Target change** (`watch(() => calc.desired …)` in `store.mjs`, deep + `flush:'sync'`) clears `selectedRecipes`; import/energy mode flags are left untouched.
- **`primaryOutput` grouping (`recipes.mjs`):** multi-output recipes are presented under a designated primary product (e.g. Sulfur Excavator → "Coal"); power recipes get a synthetic `Energy` primary output. `CANON` canonicalizes lowercase/hyphenated export codeNames to PascalCase metadata keys; `SKIP_FACILITIES` excludes Engine Rooms (their only output, ReservePower, is consumed by nothing).
- **Energy pseudo-resource:** not in `metadata.json`. Default **imported** (`calc.energyImported`, `store.mjs`); `toggleEnergy()` flips it and clears any `selectedRecipes.Energy`; assigning a power recipe (`chooseRecipe('Energy', …)`) sets `energyImported=false`. Sulfuric Reactor is a power *producer* (Sulfur is a byproduct). Net MWh ceiled to 1 decimal; negative = "produced".
- **Power model:** `effectivePower = powerDelta` (raw MW, never ÷5); grid-connected multi-order facilities run 5× faster, so the `÷5` lives on **time** (`effectiveDuration = duration/5` for consumers; producers + pads keep raw duration). `Energy MWh = effectivePower × effectiveDuration / 3.6e6`. Each recipe row shows `P MW × Ds` (PowerChip, magnitude-only, red=consumed / green=produced). Building power comes from `PowerGridInfo.PowerDelta` in the facility blueprint (`process-game-data.js`), applied to every recipe of that facility (per-recipe `powerDelta` overrides the building base; a `0` recipe inherits the building base).

## Key Data Structures

### `metadata.json` — 715 entries (245 items, 176 vehicles, 294 structures; 31 families, 90 hidden tier members, 625 searchable)

Classified into **11 behavioral classes** by `src/components/metadata-format.js` (`Mount/Deployed` currently has 0 items): `Structure` (270), `Land Vehicle` (139), `Material/Supply` (62), `Ammunition` (45), `Tool/Equip` (46), `Firearm` (50), `Misc` (29), `Ship` (24), `Aircraft` (13), `Grenade/Thrown` (10), `Melee Weapon` (3). **434** entries carry `resistances`, **389** carry a `destruction` table.

The Search detail panel renders a wiki-style **infobox** + two catch-all lists (unformatted fields, missing wiki fields).

### Field availability — what the game exports DO / DON'T contain (verified)
- ✅ Shown when present: damage / damage type / magazine / accuracy / slot / ammo / crate (weapon/ammo/itemComponent); weight (`encumbrance`, some items lack it e.g. `RifleW`); damage mult; fuze / range / explosion radius (grenades); armour HP (vehicles); health/repair/decay/storage/disable; `maxAmmo:0` is correct for melee/launchers/flamethrowers; `engineForce:0` is correct for 42 towed/relic vehicles.
- ❌ **ABSENT from all `Blueprints/Data/*` exports (wiki/live-client only — never scrape):** `reload`, `fireRate`/`FiringRate`, `firingMode` (guns), `crew`/`passengers`, ship armour/pen/trigger-mines, speed in km/h, `uses`, pallet amount, intel range, `built-with`, structure armour type. `range`/`speed`/`tankArmourMinPenetrationChance` exist but in raw game units (no conversion) → listed as "missing".
- **Resistance matrix stores a RESISTANCE FRACTION** (`0` = full damage, `1` = immune), NOT a multiplier. Effective damage = `base × (1 − fraction)`. Destruction tables (x-to-kill / disable|kill) are **merged into the Resistances pane** (`ItemDetail.vue`, single `.ktab`); upgrade families render 1+3 tier columns (Forts are the only in-metadata tier family with armourType; world-structure families like Town Base / Safe House / Relic Base are synthesized from `DTDamageProfiles` + wiki health — names/sizes are wiki-derived, values game-accurate).

### `parser/data/recipes.json` — Production Recipe Index
- `factory.crateRecipes` — Factory/MPF crate recipes (codeName → `{inputs, outputs, duration, retrieveTimes, researchLevel}`).
- `facilities.{key}` — 22 facility blueprints with `baseRecipes` + per-modification recipes.

## Operational notes (agentic workflow)
- **Loop-police:** `metadata-format.js`, `process-game-data.js`, `ItemDetail.vue` are auto-blocked from being read after ~4 reads/session. Use `grep`/`sed`/`python3` for targeted edits.
- **Semantic-loop guard:** when iterating, *act* with concrete edits, don't re-plan. If flagged as a semantic loop, stop re-deriving and apply the next concrete change.
- **Don't fully read large JSON** (`metadata.json`, `recipes.json`, game_data exports): use `jq`/`grep`/`head`/`python3` to inspect structure and spot-check.
- **Data provenance (authoritative):** every displayed value derives from exported `game_data/` via `process-game-data.js`. The foxhole wiki is **NOT** a data source — used only for testing/sanity checks (it is Cloudflare-blocked, HTTP 403, for direct fetch). Absent fields are omitted unless the user supplies the value in conversation (a "conversation exception", stored as an explicit override) — never scraped.
- **Dev-server:** restart `npm run dev` if a UI change doesn't appear; always `npm run build` to check.
- **Small implementations; minimal comments.** Don't dump full example files into comments — rely on the example CSVs / `data/` files. Prefer `python3`/shell one-offs in `tmp/` for research.

## Known data quirks (verified against game_data exports)
- **Handheld weapon reload is NOT in game exports** (`ReloadTime` in 0 files; only `mountData.reloadDuration` for deployed weapons). Omit — do not re-add from the wiki.
- **Vehicle speed is a physics-sim output, not a scalar.** `defaultSurfaceMovementRate` is normalized 0..1; a linear fit from `engineForce/mass/airResistance/rollingResistance` reaches only R²≈0.53. Aircraft/ship speed lives only in `airData`/`shipData`.
- **5 firearms lack `encumbrance`** in exports (SMGHeavyW/C, RifleW, RifleLightC, RifleAutomaticW) → rendered gracefully as absent.
- **`maxAmmo:0`** correct for 9 melee/launchers + flamethrowers (fuel tank, not a magazine).
- **`engineForce:0`** correct for 42 towed/relic vehicles (pulled, not self-propelled).
- **Powerless buildings (source data):** Water Pump, base Oil Well, the resource harvesters (Salvage/Components/Sulfur/Coal mines), Offshore Platform, and Concrete Mixer all have `0` building power. (User once believed only Water Pump + base Oil Well lack power — trust the data.)
- **Factory/MPF `crateRecipes` must be gated by `productionCategories`** (factory/massProduction queue type). Many facility-only items carry a legacy `CostPerCrate` but are NOT Factory/MPF products — listing them there is the (fixed) bug where e.g. Cmats appeared makeable at a Factory/MPF. `isMpfEligible` excludes any vehicle with a facility-out recipe; vehicles/structures correctly have `productionCategories:null` yet ARE MPF-eligible via build cost — do **not** gate those by it.
- **Battleship casing:** Dry Dock recipes use `LargeShipBattleShip*` (one 's') while metadata/icon is `LargeShipBattleship*` (canonicalized in `recipes.mjs`). Dry Dock recipes have empty inputs in the export — faithful, NOT a bug.
